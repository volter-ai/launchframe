## Patterns To Follow

- Use a plain title that names the artifact and the specific user problem.
- Start the comment with why the builder needed it.
- Explain the mechanism quickly.
- Include one concrete command or demo path.
- Say what is rough or early instead of pretending polish.
- Ask for a specific kind of feedback from the right audience.
- Be ready for skeptical comparison questions.

## Patterns To Avoid

- Marketing language without a mechanism.
- Claims like "no tracking" unless every script/network path supports the claim.
- Posting from a brand-new HN account immediately after creation.
- Asking for generic support or votes.
- Overexplaining business model before the technical value is clear.

## Likely HN Questions

- Why not just use a checklist in the repo?
- Why not a GitHub issue template or project board?
- Is this just process theater around launching?
- What does the CLI validate beyond file presence?
- What is the smallest successful launch this has handled?
- Is any of this automated posting or vote-seeking?

## Copy Implication

The Show HN comment should be more concrete and less grand. Lead with the artifact: a CLI that creates a launch workspace and validates required launch files. Then explain that the value is reconstructability after agent-assisted work, not launch automation.
